# Strava API notes
## Strava API V3 OAuth 2.0 Authentication Architecture
The Strava V3 API mandates the use of OAuth 2.0 to facilitate secure delegation of access to its athletic dataset. This architecture operates on a token-based paradigm, utilizing a multi-step sequence:
- Browser-based user authorization.
- Authorization code exchange.
- Issuance of short-lived bearer tokens paired with mutable refresh tokens.

## The Authorization Phase: Initiating the OAuth Flow
The authentication sequence is initiated when a user attempts to connect their Strava account to a third-party application. Because the application fundamentally does not possess or handle the user's primary credentials, it must orchestrate a process to redirect the user's user-agent (typically a web browser or a system-level webview) to Strava's dedicated authorization endpoint.
For traditional web applications accessed via desktop or mobile browsers, the application must issue a redirect to the standard web endpoint located at GET `https://www.strava.com/oauth/authorize`. Conversely, applications running natively on mobile operating systems must utilize the dedicated mobile endpoint located at GET `https://www.strava.com/oauth/mobile/authorize` to ensure the correct presentation and handling of the native application context.

Parameters:
- client_id	(Required) The integer-based application identifier obtained during the developer registration process. This tells Strava which app is initiating the prompt.
- redirect_uri (Required) The specific destination URL to which the user-agent will be redirected immediately after the authorization interaction concludes. It must fall strictly within the registered callback domain.
- response_type	(Required) This value must be statically hardcoded to the string code, explicitly indicating to the Strava server that the application is utilizing the OAuth 2.0 Authorization Code Grant type.
- scope	(Required) A comma-delimited list of strings specifying the exact granular permissions the application is requesting (e.g., read, activity:write).
- approval_prompt (Optional) Accepts the string values auto (the default behavior) or force. Setting this to force ensures the consent screen is visually presented to the user every single time, even if they have previously authorized the application.
- state	(Optional) An arbitrary, cryptographically secure random string generated by the application. Strava promises to return this exact value during the redirection phase. It is fundamentally critical for tracking application state and mitigating Cross-Site Request Forgery (CSRF) attacks.

### Scope Taxonomy and the Principle of Least Privilege
Strava uses a hierarchical taxonomy of permissions. Users can selectively opt out of specific scopes during consent.
- read Public segments, profiles, and activities.
- read_all Includes private routes, segments, and events.
- profile:read_all Full profile access regardless of privacy settings.
- activity:read Public activities; excludes data in user privacy zones.
- activity:read_all Full activity access, including raw geospatial data in privacy zones.
- activity:write Programmatic creation/upload and editing of activities.

### Platform-Specific Implementation
The server issues an HTTP 302 redirect to the authorization URL.

### The Token Exchange Paradigm
The `code` returned via redirect is a single-use, ephemeral credential. It must be exchanged server-side for an access token.

## POST Request Formulation
Endpoint: `https://www.strava.com/oauth/token`
```
curl -X POST https://www.strava.com/api/v3/oauth/token \
  -d client_id=[ID] \
  -d client_secret=[SECRET] \
  -d code=[AUTHORIZATION_CODE] \
  -d grant_type=authorization_code
```
JSON Response Payload
- access_token (String) The bearer token for API requests.
- expires_at (Integer) Unix epoch timestamp of expiration.
- refresh_token (String) Used to obtain new access tokens.
- athlete (Object) Summary profile data of the authenticated user.

### Token Management and Lifecycle
Strava enforces a dynamic lifecycle to minimize the impact of credential theft.
- Expiration: Access tokens expire after 6 hours (21,600 seconds).
- Refresh Protocol: Use the refresh_token to request a new access_token via POST without re-prompting the user.
- Token Overlap: If refreshing within one hour of expiration, both the old and new tokens remain valid until their respective expires_at times.
- Refresh Token Mutation: The refresh_token may change during a refresh event. Applications must update their database with the new refresh token immediately, as the old one is invalidated.
